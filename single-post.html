Getting Started with POS Hardware Integration on Android

Overview

Integrating Android applications with POS hardware is one of those tasks that sounds straightforward but hides a surprising amount of complexity. Between USB serial communication, vendor-specific SDKs, and the variety of Android OS versions running on POS terminals in the wild, there's a lot to navigate.

This guide covers the core concepts and practical steps I've learned from shipping POS integrations in production, with a focus on getting reliable communication between your Android app and the attached hardware peripherals.

Prerequisites

Before diving in, make sure you have:

  • An Android development environment set up (Android Studio / VS Code with Flutter)
  • Access to the POS terminal's SDK or documentation from the hardware vendor
  • A physical device to test on — emulators won't cut it for hardware communication
  • Basic familiarity with Android permissions and USB communication

Serial Communication

Most POS terminals expose their peripherals (card readers, receipt printers, barcode scanners) over a serial interface. On Android, you have two main approaches:

  1. USB Serial via USB Host API — works when the Android device acts as the USB host. You enumerate connected USB devices, find your device by vendor/product ID, and open a serial port.
  2. Vendor SDK — most POS manufacturers ship an SDK that wraps the serial communication behind a cleaner API. Always prefer this when available.

When working with the USB Host API directly, declare the required permission in your manifest and handle the permission request at runtime before attempting to open the connection.

AndroidManifest.xml
<uses-feature android:name="android.hardware.usb.host" />

<uses-permission android:name="android.permission.USB_PERMISSION" />

Receipt Printing

Receipt printers in POS terminals typically understand ESC/POS commands — a set of escape sequences that control formatting, alignment, and paper cutting. Most vendor SDKs expose a higher-level print API, but understanding ESC/POS helps when debugging output issues.

Key things to handle reliably:

  • Check printer status before sending a job (paper out, cover open, etc.)
  • Queue print jobs and handle failures with user-facing feedback
  • Test with real receipt paper at different temperatures — thermal printing degrades on cheap paper
  • Never block the main thread during print operations; always use a background thread or coroutine

Tips & Gotchas

After shipping several POS integrations, these are the things I wish I'd known upfront:

  • Device fragmentation is real. The same app may behave differently across POS terminal brands even when they claim Android compatibility. Always test on the exact hardware you're shipping on.
  • USB permissions don't persist across reboots on some devices. Handle permission denial gracefully and re-request when the app relaunches.
  • Use a connection watchdog. Serial connections can silently drop. Implement a heartbeat or reconnect logic so the app recovers automatically.
  • Log everything during development. Hardware bugs are hard to reproduce. Detailed logs of bytes sent/received save hours of debugging.

POS integrations reward careful, defensive coding. Once you've shipped one, you'll have a solid mental model that makes every subsequent integration much faster.
